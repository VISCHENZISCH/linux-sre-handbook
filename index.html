Administration Serveur Linux

Un manuel d'ingénierie système moderne, de sécurité et d'observabilité. Conçu avec la sobriété et la rigueur d'un outil de développement de premier ordre.

~35 min de lecture Tous les Niveaux Mis à jour Mai 2026

Objectif général

Ce guide couvre l'intégralité du cycle de vie d'un serveur Linux : de son identification à sa mise en production, en passant par la maintenance quotidienne, la gestion technique et les pratiques DevOps modernes.

Les commandes sont divisées en catégories :

Bleu — La finalité du serveur : ports exposés et services actifs.
Rouge — Le niveau de saturation des ressources matérielles.
Vert — Les erreurs potentielles et événements critiques.

Types de serveurs Linux

Un serveur Linux peut remplir une multitude de rôles. Voici les typologies les plus courantes rencontrées en entreprise et en infrastructure cloud.

Serveur Web 80, 443

Héberge des sites et applications. Utilise Nginx, Apache ou LiteSpeed. Souvent le point d'entrée principal du trafic.

Nginx Apache

Base de Données 5432, 3306

Stocke et gère les données structurées. Critique pour les performances applicatives. Nécessite un tuning kernel spécifique.

PostgreSQL MariaDB

Reverse Proxy / LB 80, 443

Répartit la charge entre plusieurs serveurs backend et gère la terminaison SSL/TLS.

HAProxy Traefik

Serveur de Fichiers 2049, 445

Partage de ressources de stockage sur le réseau local ou étendu. Utilise NFS ou Samba (SMB/CIFS).

NFS Samba

Serveur DNS 53

Résolution de noms de domaine en adresses IP. Élément fondamental de toute infrastructure réseau.

BIND9 Unbound

Serveur Mail 25, 587, 993

Gère l'envoi (SMTP) et la réception (IMAP/POP3) de courriels. Requiert une configuration antispam rigoureuse.

Postfix Dovecot

01 · Caractériser le serveur

Identifier la mission du serveur en analysant les ports ouverts et les processus qui les écoutent.

Commandes de diagnostic réseau 
admin@srv:~$ sudo ss -tlpn        # ports en écoute (moderne)
admin@srv:~$ sudo netstat -tlpn   # ancienne commande
admin@srv:~$ sudo ps auxf         # processus en arbre

Exemple de résultat analysé :

Proto  Local Address       State   PID/Program
tcp    0.0.0.0:22          LISTEN  601/sshd
tcp    127.0.0.1:5432      LISTEN  625/postgres
tcp    0.0.0.0:8000        LISTEN  885/haproxy
Analyse : Le serveur expose un service SSH (22) et un HAProxy (8000). Il fait donc office de Load Balancer. Une base PostgreSQL tourne localement.

02 · Ressources matérielles

Mesurer la charge du système (CPU, RAM, Disque) pour identifier les goulots d'étranglement.

CPU & Mémoire

admin@srv:~$ htop
admin@srv:~$ uptime
admin@srv:~$ free -m

Stockage I/O

admin@srv:~$ df -h
admin@srv:~$ lsblk
admin@srv:~$ sudo iotop
Alerte OOM : Si vous voyez un processus tué mystérieusement, vérifiez le kernel log : dmesg | grep -i "out of memory".

03 · Explorer les applications

Comprendre l'interaction entre les différents services installés (Systemd, Docker, scripts).

Client ➜ HAProxy (:8000) ➜ Webapp Python (:9000) ➜ PostgreSQL (:5432) 
admin@srv:~$ sudo systemctl status webapp
admin@srv:~$ docker ps -a
admin@srv:~$ cat /home/admin/.env

04 · Analyser les logs

La source ultime de vérité pour diagnostiquer les erreurs applicatives et système.

admin@srv:~$ sudo journalctl -u webapp -f
admin@srv:~$ sudo journalctl -p err --since "1 hour ago"
admin@srv:~$ tail -f /var/log/syslog

Sécurité & Hardening

Réduire la surface d'attaque par le durcissement système et applicatif.

SSH Hardening

# /etc/ssh/sshd_config
PermitRootLogin no
PasswordAuthentication no
Port 2222
AllowUsers deployer admin

Firewall (UFW)

# ufw default deny incoming
# ufw allow 2222/tcp
# ufw allow 80,443/tcp
# ufw enable

Fail2Ban

# apt install fail2ban
# Protection contre le brute force
# fail2ban-client status sshd

Packages & Dépendances

Gestion propre des logiciels et des mises à jour de sécurité.

# apt update && apt upgrade -y
# apt-mark hold nginx  # Bloquer la version
# apt autoremove --purge

Systemd Avancé

Maîtriser les unités de service pour une résilience maximale.

Exemple d'Unit File Robuste 
# /etc/systemd/system/webapp.service
[Unit]
Description=Webapp Python
After=network.target postgresql.service

[Service]
User=webapp
Restart=on-failure
RestartSec=5s
CapabilityBoundingSet=CAP_NET_BIND_SERVICE
PrivateTmp=true

[Install]
WantedBy=multi-user.target

Maintenance planifiée

Une maintenance rigoureuse prévient les pannes critiques et assure la longévité de l'infrastructure.

Quotidien

  • Vérification des sauvegardes
  • Analyse des logs d'erreurs critiques
  • Surveillance des alertes monitoring

Hebdomadaire

  • Mises à jour de sécurité (staging)
  • Rapport d'utilisation des ressources
  • Nettoyage des fichiers temporaires

Mensuel

  • Test de restauration des backups
  • Audit de sécurité (Lynis)
  • Optimisation des bases de données

Gestion technique

L'administration efficace passe par une documentation à jour et des procédures standardisées.

CMDB & Inventaire

Gardez une trace de chaque machine : specs, rôle, propriétaire technique et criticité.

Sauvegarde & PRA/PCA

La règle d'or : 3-2-1 (3 copies, 2 supports, 1 hors-site).

RTO (Recovery Time Objective)

Temps maximum acceptable pour rétablir le service après une panne.

RPO (Recovery Point Objective)

Perte de données maximale acceptable (ex: 1 heure de données).

Docker & Conteneurs

Isolation applicative légère. Docker permet de packager une application et ses dépendances dans une image immuable.

Workflow Docker

  • Build : Créer une image via un Dockerfile
  • Ship : Stocker sur un Registry (Docker Hub)
  • Run : Instancier un conteneur partout

Commandes Essentielles

# docker build -t myapp .
# docker run -d -p 80:80 myapp
# docker exec -it myapp sh
# docker network ls

Docker Compose Avancé

# docker-compose.yml
version: "3.9"
services:
  backend:
    build: ./api
    environment:
      - DATABASE_URL: postgres://user:pass@db:5432/db
    depends_on:
      - db
  db:
    image: postgres:15-alpine
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:

Haute Disponibilité

Éliminer les points de défaillance uniques (SPOF) garantit qu'aucune panne d'un composant isolé ne peut interrompre le service. La haute disponibilité repose sur trois piliers : la redondance, le basculement automatique et la réplication des données.

Keepalived / VIP

Keepalived utilise le protocole VRRP pour faire flotter une IP virtuelle (VIP) entre deux nœuds. Si le nœud primaire devient inaccessible, le secondaire prend automatiquement l'IP en moins d'une seconde.

# apt install keepalived
# systemctl enable --now keepalived
# ip addr show  # vérifier la VIP active

Réplication de BDD

La réplication en streaming de PostgreSQL envoie chaque transaction du nœud primaire vers un standby. En cas de panne, une promotion automatique (Patroni) rend le standby opérationnel.

# pg_basebackup -h primary -D /data -P -Xs -R
# patronictl failover cluster --master primary
Règle d'or HA : Testez votre basculement en production planifiée avant qu'il arrive en urgence. Un failover non testé est un failover qui échoue au pire moment.

Checklist SPOF

  • Load Balancer : 2 nœuds en actif/passif (Keepalived)
  • Base de données : primaire + au moins 1 réplica
  • Stockage : RAID logiciel (mdadm) ou volume distribué (Ceph)
  • Réseau : 2 interfaces en bonding (mode active-backup ou LACP)

Automatisation & IaC

Traiter l'infrastructure comme du code signifie que chaque serveur, réseau et règle de firewall est décrit dans un fichier versionné dans Git. L'environnement devient reproductible et auditable.

Terraform (Provisioning)

Terraform déclare les ressources cloud dans des fichiers .tf. Il calcule un plan d'exécution avant d'appliquer les changements.

# terraform plan  # prévisualiser
# terraform apply # appliquer

Ansible (Configuration)

Ansible configure les serveurs sans agent via SSH. Les playbooks YAML sont idempotents : parfaits pour le hardening et les mises à jour.

# ansible-playbook -i hosts site.yml
# ansible all -m ping

Packer (Golden Images)

Packer crée des images machine immuables (AMI, QCOW2) préconfigurées. On démarre directement depuis une image déjà durcie et testée.

Ordre recommandé : Packer construit l'image ➜ Terraform provisionne les instances ➜ Ansible finalise la configuration applicative.

Cloud & Hybride

L'approche hybride combine infrastructure on-premise et services cloud managés pour exploiter l'élasticité tout en gardant le contrôle.

Les trois modèles

IaaS : vous gérez l'OS (EC2, Compute Engine).
PaaS : provider gère l'OS (Heroku, App Service).
SaaS : clé en main (RDS, BigQuery).

Connectivité Hybride

Utilisez un VPN IPSec ou une liaison privée dédiée (Direct Connect, ExpressRoute) pour une latence prévisible et sécurisée.

Bonne pratique : Préférer des services ouverts (PostgreSQL RDS) pour éviter le vendor lock-in et versionner toute l'infra dans Git.

Performance & Tuning

Optimisation du kernel et des paramètres système (sysctl).

# /etc/sysctl.d/99-perf.conf
vm.swappiness = 10
net.core.somaxconn = 65535
fs.file-max = 2097152

Dépannage Avancé

Face à une panne, l'improvisation coûte du temps. La méthode USE (Utilization, Saturation, Errors) et la méthode RED (Rate, Errors, Duration) sont deux cadres de diagnostic systématiques qui permettent d'isoler la cause racine en moins de 10 minutes.

Méthode USE

Utilization : Capacité utilisée (htop, free).
Saturation : File d'attente (load average).
Errors : Erreurs kernel (dmesg).

Méthode RED

Rate : Requêtes par seconde.
Errors : Pourcentage d'échecs.
Duration : Latence (P95, P99).

CoucheSymptômeOutil
CPULoad average élevétop, perf top
MémoireOOM killsdmesg, vmstat
DisqueI/O wait > 20%iostat, iotop
RéseauPaquets perdusss -s, tcpdump
 
# dmesg -T | tail -50                    # événements kernel
# journalctl -p err -b                   # erreurs boot
# ss -tnp state established              # connexions actives
# strace -p <PID> -e trace=network       # appels réseau
En crise : Avant de redémarrer quoi que ce soit, capturez l'état du système (ps auxf, netstat, dmesg). Un redémarrage efface les preuves nécessaires pour éviter la prochaine panne.

SRE & Fiabilité

Le Site Reliability Engineering applique l'ingénierie logicielle aux problèmes d'exploitation. L'objectif est de quantifier la fiabilité acceptable, automatiser les tâches répétitives (toil) et apprendre de chaque incident.

SLI, SLO, SLA

SLI : Métrique mesurée.
SLO : Cible interne (ex: 99.9%).
SLA : Engagement contractuel.

Error Budget

Tolérance à la panne autorisée par le SLO. À 99,9% : budget de 43 min/mois. Si consommé, on gèle les déploiements.

Post-mortem blameless : Analyse de l'incident sans recherche de coupable. Chronologie, cause racine (5 pourquoi) et actions correctives.

GitOps & CI/CD

GitOps fait de Git la source unique de vérité pour l'état de l'infrastructure. Tout changement passe par une Pull Request, est testé automatiquement, puis synchronisé par un agent.

ArgoCD (GitOps Pull-based)

ArgoCD surveille un dépôt Git et synchronise automatiquement le cluster Kubernetes avec l'état déclaré.

# argocd app list
# argocd app sync mon-app
# argocd app rollback mon-app 3

Canary5% du trafic sur la nouvelle versionTrès faible

DevSecOps

Intégrer la sécurité à chaque étape du cycle de développement (Shift Left). Plus une vulnérabilité est détectée tôt, moins elle coûte à corriger.

Gestion des Secrets

Ne stockez jamais de secrets en clair dans Git. Utilisez HashiCorp Vault, AWS Secrets Manager ou Sealed Secrets.

Conformité

Utilisez Lynis pour auditer le hardening système et OpenSCAP pour vérifier la conformité aux benchmarks CIS.

Bénéfices IDP

  • Onboarding service : 2 sem ➜ 1 jour
  • Élimination des tickets ops répétitifs
  • Standardisation sans friction

FinOps

Gestion financière du cloud pour maximiser la valeur de chaque euro dépensé. Sans FinOps actif, une facture cloud peut doubler en 6 mois.

Optimisation

Right-sizing : redimensionner les instances.
Reserved Instances : s'engager sur 1 ou 3 ans (-60%).
Spot : instances interruptibles (-90%).

Alertes budgétaires

Configurez des alertes à 50%, 80% et 100% du budget mensuel. Ne découvrez jamais une anomalie en fin de mois.

Gaspillages fréquents

  • Volumes EBS non attachés
  • Snapshots anciens (> 90 jours)
  • IP élastiques réservées non associées
  • Instances de dev tournant le week-end

Observabilité avancée

L'observabilité permet de comprendre n'importe quel état interne du système à partir de ses sorties externes — y compris les pannes qu'on n'avait pas anticipées.

Les trois piliers

Métriques : Prometheus (Quoi).
Logs : Loki (Pourquoi).
Traces : Tempo (Où).

eBPF & Profiling

Observer le kernel sans modifier le code (Cilium). Le profiling continu (Pyroscope) montre quelles lignes de code consomment le plus.

Stack LGTM : Loki + Grafana + Tempo + Mimir. Un écosystème cohérent et open-source pour une visibilité totale.

Orchestration Kubernetes

Le standard de l'industrie pour orchestrer des conteneurs à grande échelle. Automatise le déploiement, le scaling et la gestion opérationnelle.

Control Plane

Le cerveau du cluster : API Server, Scheduler, et l'état désiré via etcd.

Worker Nodes

Où tournent les conteneurs (Pods). Gérés par le Kubelet et kube-proxy.

Commandes kubectl Critiques

# Diagnostic global
# kubectl get pods -A -o wide
# kubectl top nodes
# Debug direct
# kubectl describe pod <pod_name>
# kubectl logs -f <pod_name> --tail=100

Entretiens Techniques (Q&R)

Questions de filtre pour postes DevOps / SRE / Cloud / Infrastructure. La plupart sont éliminatoires.

Partie 1 — Réseau

Explique le three-way handshake TCP

1. SYN : Client envoie SYN (numéro de séquence initial).
2. SYN + ACK : Serveur acquitte le SYN client et envoie le sien.
3. ACK : Client répond ACK. Connexion établie.

Le champ SYN synchronise les numéros de séquence pour tracker les octets. ACK indique le prochain octet attendu.

Différence entre TCP et UDP ?

TCP : Orienté connexion, garantit la livraison (retransmission), vérifie les erreurs. Plus lent. (HTTP, SSH, SMTP).

UDP : Sans connexion (fire & forget), aucune garantie, supporte le broadcasting. Plus rapide. (DNS, VoIP, Streaming).

DNS Records

  • A : IPv4 | AAAA : IPv6 (::1 = localhost)
  • CNAME : Alias | MX : Mail
  • PTR : Reverse | TXT : SPF/DKIM

Ports & Codes HTTP

  • 22: SSH | 53: DNS | 443: HTTPS
  • 200: OK | 403: Forbidden | 404: Not Found
  • 502: Bad Gateway | 504: Timeout

Partie 2 — Système Linux

Qu'est-ce qu'un inode ?

Structure de données stockant les métadonnées (Type, Perms, UID/GID, Taille, Timestamps). Ne contient pas le nom du fichier.

Types : (-) fichier, (d) répertoire, (l) lien, (b/c) device, (p) pipe, (s) socket.

Permissions Spéciales (SUID, SGID, Sticky)

SUID (s) : S'exécute avec les droits du proprio (ex: passwd).
SGID (s) : Hérite du groupe du parent.
Sticky bit (t) : Seul le proprio peut supprimer (ex: /tmp).

Partie 3 — Processus

États des processus Linux

R : Running | S : Sleeping (interruptible)
D : Uninterruptible (I/O) - ne peut pas être tué par SIGKILL
Z : Zombie (terminé, attend wait() du parent)
T : Stopped (Ctrl+Z)

Peut-on tuer un zombie ?

Non, il est déjà mort. Pour le faire disparaître, il faut tuer son processus parent pour que init (PID 1) récupère l'enfant et appelle wait().

Signaux non ignorables ?

Seuls SIGKILL (9) et SIGSTOP (19) ne peuvent pas être ignorés ou interceptés par le processus.

À quoi sert strace ? 
# strace -p <PID>             # Attacher à un processus
# strace -e trace=network cmd  # Filtrer les appels réseau

Trace tous les appels système (syscalls) en temps réel pour déboguer fichiers ouverts, erreurs réseau, etc.

Partie 4 — Autres & Stratégie

Que se passe-t-il pour ls *.txt ?

1. Le shell effectue le glob expansion (*.txt ➜ liste de fichiers).
2. Le shell remplace l'argument par la liste résolue.
3. ls est exécuté avec les noms de fichiers réels. ls ne voit jamais l'étoile.

Kernel & Système

modprobe : Charger module + dépendances.
ulimit : Limites de ressources (fichiers ouverts, etc).
sysctl : Paramètres kernel à chaud (ex: ip_forward).
