Diagnostic & Base

Méthodes et outils pour analyser les services, le hardware et les logs d'un serveur Linux en temps réel.

~10 min de lecture Pratique / Sysadmin Mis à jour Mai 2026

01 · Caractériser le serveur

Identifier la mission du serveur en analysant les ports ouverts et les processus qui les écoutent.

Commandes de diagnostic réseau 
admin@srv:~$ sudo ss -tlpn        # ports en écoute (moderne)
admin@srv:~$ sudo netstat -tlpn   # ancienne commande
admin@srv:~$ sudo ps auxf         # processus en arbre

Exemple de résultat analysé :

Proto  Local Address       State   PID/Program
tcp    0.0.0.0:22          LISTEN  601/sshd
tcp    127.0.0.1:5432      LISTEN  625/postgres
tcp    0.0.0.0:8000        LISTEN  885/haproxy
Analyse : Le serveur expose un service SSH (22) et un HAProxy (8000). Il fait donc office de Load Balancer. Une base PostgreSQL tourne localement.

02 · Ressources matérielles

Mesurer la charge du système (CPU, RAM, Disque) pour identifier les goulots d'étranglement.

CPU & Mémoire

admin@srv:~$ htop
admin@srv:~$ uptime
admin@srv:~$ free -m

Stockage I/O

admin@srv:~$ df -h
admin@srv:~$ lsblk
admin@srv:~$ sudo iotop
Alerte OOM : Si vous voyez un processus tué mystérieusement, vérifiez le kernel log : dmesg | grep -i "out of memory".

03 · Explorer les applications

Comprendre l'interaction entre les différents services installés (Systemd, Docker, scripts).

Client ➜ HAProxy (:8000) ➜ Webapp Python (:9000) ➜ PostgreSQL (:5432) 
admin@srv:~$ sudo systemctl status webapp
admin@srv:~$ docker ps -a
admin@srv:~$ cat /home/admin/.env

04 · Analyser les logs

La source ultime de vérité pour diagnostiquer les erreurs applicatives et système.

admin@srv:~$ sudo journalctl -u webapp -f
admin@srv:~$ sudo journalctl -p err --since "1 hour ago"
admin@srv:~$ tail -f /var/log/syslog

Diagnostic Rapide en Production (Crisis Management)

En tant que SRE, lorsqu'une alerte retentit, vous devez diagnostiquer la cause racine (CPU, RAM, Disque, Réseau) sous pression. Suivez cette checklist en 5 minutes :

1. Tri CPU & RAM

Tapez top ou htop. Triez par CPU (%) avec P ou par mémoire (%) avec M. Repérez le processus incriminé.

2. Vérification I/O Disque

Tapez iostat -xz 1 10. Si %util approche 100% ou si await dépasse 20ms, vos disques saturent. Identifiez le coupable avec iotop -o.

3. Analyse Réseau

Tapez ss -s pour voir le nombre total de sockets TCP actives. Utilisez iftop pour détecter une consommation anormale de bande passante par adresse IP.

Pratique & Apprentissage Interactif

Pour vous entraîner sur de vrais incidents de production Linux dans un environnement sécurisé et interactif :

  • SadServers — Résolution d'incidents réels sur des serveurs Linux (scénarios de CPU saturés, disques pleins, sockets perdues).
  • The Linux Documentation Project (TLDP) — Guides complets d'administration système et réseaux.
