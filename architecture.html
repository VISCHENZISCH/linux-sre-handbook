Docker & Conteneurs

Isolation applicative légère. Docker permet de packager une application et ses dépendances dans une image immuable.

Workflow Docker

  • Build : Créer une image via un Dockerfile
  • Ship : Stocker sur un Registry (Docker Hub)
  • Run : Instancier un conteneur partout

Commandes Essentielles

# docker build -t myapp .
# docker run -d -p 80:80 myapp
# docker exec -it myapp sh
# docker network ls

Docker Compose Avancé

# docker-compose.yml
version: "3.9"
services:
  backend:
    build: ./api
    environment:
      - DATABASE_URL: postgres://user:pass@db:5432/db
    depends_on:
      - db
  db:
    image: postgres:15-alpine
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:

🐳 Dockerfile de Production Optimisé & Sécurisé

Ce patron de conception applique les meilleures pratiques de sécurité et d'optimisation (multi-stage builds, utilisateur non-root, cache de dépendances, et réduction des couches d'image) :

# ── ÉTAPE 1 : CONSTRUIRE (Builder) ──
FROM node:20.12.0-alpine3.19 AS builder
WORKDIR /app

# Copier les fichiers de dépendances en premier pour maximiser l'utilisation du cache Docker
COPY package.json package-lock.json ./
RUN npm ci --silent

# Copier le code source de l'application et compiler
COPY . .
RUN npm run build

# ── ÉTAPE 2 : EXÉCUTER (Runtime minimal) ──
FROM node:20.12.0-alpine3.19 AS runner
WORKDIR /app

# Définir l'environnement de production
ENV NODE_ENV=production

# Sécurité : Création d'un groupe et d'un utilisateur non-root (évite l'élévation de privilèges)
RUN addgroup -S nodeapp && adduser -S nodeuser -G nodeapp

# Récupérer UNIQUEMENT les fichiers compilés du Builder (réduit la taille de l'image finale)
COPY --from=builder --chown=nodeuser:nodeapp /app/dist ./dist
COPY --from=builder --chown=nodeuser:nodeapp /app/package.json /app/package-lock.json ./

# Installer uniquement les dépendances de production
RUN npm ci --only=production --silent

# Basculer vers l'utilisateur non-privilégié
USER nodeuser

EXPOSE 3000
CMD ["node", "dist/index.js"]

Haute Disponibilité (High Availability)

Éliminer les points de défaillance uniques (SPOF) garantit qu'aucune panne d'un composant isolé ne peut interrompre le service. La haute disponibilité repose sur trois piliers : la redondance, le basculement automatique et la réplication des données.

Keepalived / VIP

Keepalived utilise le protocole VRRP pour faire flotter une IP virtuelle (VIP) entre deux nœuds. Si le nœud primaire devient inaccessible, le secondaire prend automatiquement l'IP en moins d'une seconde.

# apt install keepalived
# systemctl enable --now keepalived
# ip addr show  # vérifier la VIP active

Réplication de BDD

La réplication en streaming de PostgreSQL envoie chaque transaction du nœud primaire vers un standby. En cas de panne, une promotion automatique (Patroni) rend le standby opérationnel.

# pg_basebackup -h primary -D /data -P -Xs -R
# patronictl failover cluster --master primary
Règle d'or HA : Testez votre basculement en production planifiée avant qu'il arrive en urgence. Un failover non testé est un failover qui échoue au pire moment.

Checklist SPOF

  • Load Balancer : 2 nœuds en actif/passif (Keepalived)
  • Base de données : primaire + au moins 1 réplica
  • Stockage : RAID logiciel (mdadm) ou volume distribué (Ceph)
  • Réseau : 2 interfaces en bonding (mode active-backup ou LACP)

Automatisation & IaC

Traiter l'infrastructure comme du code signifie que chaque serveur, réseau et règle de firewall est décrit dans un fichier versionné dans Git. L'environnement devient reproductible et auditable.

Terraform (Provisioning)

Terraform déclare les ressources cloud dans des fichiers .tf. Il calcule un plan d'exécution avant d'appliquer les changements.

# terraform plan  # prévisualiser
# terraform apply # appliquer

Ansible (Configuration)

Ansible configure les serveurs sans agent via SSH. Les playbooks YAML sont idempotents : parfaits pour le hardening et les mises à jour.

# ansible-playbook -i hosts site.yml
# ansible all -m ping

Packer (Golden Images)

Packer crée des images machine immuables (AMI, QCOW2) préconfigurées. On démarre directement depuis une image déjà durcie et testée.

Ordre recommandé : Packer construit l'image ➜ Terraform provisionne les instances ➜ Ansible finalise la configuration applicative.

⚙️ Configuration SRE de Terraform : Remote Backend & Lock

Pour collaborer de manière sécurisée en équipe sans risquer d'écraser des ressources ou de corrompre le fichier d'état (terraform.tfstate), utilisez un backend distant sécurisé. Cette configuration utilise la **nouvelle fonctionnalité native de locking S3** de Terraform 1.10+ (sans nécessiter de table DynamoDB externe) :

# backend.tf
terraform {
  required_version = ">= 1.10.0"

  required_providers = {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }

  # Configuration du stockage d'état sécurisé et du verrouillage (State Locking)
  backend "s3" {
    bucket         = "company-sre-terraform-state"
    key            = "production/vpc-and-compute.tfstate"
    region         = "eu-west-3"  # Paris
    
    # Sécurité & Intégrité
    encrypt        = true  # Chiffrement à l'état de repos (AES-256)
    
    # Nouveau standard Terraform 1.10+ : Verrouillage natif via fichier S3
    use_lockfile   = true  # Élimine le besoin d'une table DynamoDB tierce
  }
}
Bonne pratique SecOps : Ne commitez jamais les fichiers de statut locaux (.tfstate) dans Git. Ils contiennent des secrets et des mots de passe d'infrastructure en clair. Ajoutez-les systématiquement à votre .gitignore.

Cloud & Hybride

L'approche hybride combine infrastructure on-premise et services cloud managés pour exploiter l'élasticité tout en gardant le contrôle.

Les trois modèles

IaaS : vous gérez l'OS (EC2, Compute Engine).
PaaS : provider gère l'OS (Heroku, App Service).
SaaS : clé en main (RDS, BigQuery).

Connectivité Hybride

Utilisez un VPN IPSec ou une liaison privée dédiée (Direct Connect, ExpressRoute) pour une latence prévisible et sécurisée.

Bonne pratique : Préférer des services ouverts (PostgreSQL RDS) pour éviter le vendor lock-in et versionner toute l'infra dans Git.
